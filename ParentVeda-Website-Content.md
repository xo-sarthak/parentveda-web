### Functional spec (for the developer)

- **One endpoint:** `POST /api/subscribe` with `{ email, waitlist: boolean, newsletter: boolean, source: string }`.
  - `source` = which form/page it came from (`hero`, `waitlist-section`, `footer`, `journal-post:<slug>`, `ask-veda`) — useful for analytics.
- **At least one box must be checked.** If a user unchecks both, disable submit (or default `newsletter = true`).
- **The "newsletter-only" form** posts `{ waitlist: false, newsletter: true }`.
- **One unified email list** in the email provider, with **tags**: `waitlist`, `newsletter`, and the `source`. (Do **not** build two disconnected lists — the Ask Veda form, the WhatsApp opt-in's email, the footer, and this module all feed the same tagged list.)
- **Double opt-in** (confirmation email) — better deliverability + DPDP/consent hygiene. Success state: *"Almost there — check your inbox for a gentle confirmation. 🌸"*
- **States:** idle · submitting · success · already-subscribed (friendly: *"You're already on the list — we'll see you soon."*) · error (calm retry).
- **Spam protection:** honeypot field + basic rate-limit on the API route. No CAPTCHA (kills the calm vibe).
- **Privacy:** link "your email stays private" → `/privacy`. Don't pre-tick anything beyond the two intent boxes; no marketing-partner sharing.
- **Provider recommendation:** something with tagging + a real newsletter editor + good India deliverability. Good fits: **Brevo** (free tier, India-friendly), **Kit/ConvertKit** (creator-grade newsletters + tags), or **Resend Audiences + Broadcasts** (most dev-native with Next.js). Pick one; the API route abstracts it so you can swap later.
